# 用户分组首选渠道设计

> 状态：已实现。

## 1. 目标

系统是纯上游对接网关，允许用户为自己可用的分组各选择一个首选渠道；未选中的渠道继续使用管理员现有的全局路由规则。

已确定的业务要求：

- 配置绑定用户，而不是 API Token。
- 每个分组最多配置一个首选渠道。
- `auto` 分组不允许配置首选渠道。
- 分组列表复用创建 API 密钥时显示的分组列表。
- 用户只看到当前启用的渠道。
- 所有启用渠道默认可供用户选择，不增加“公开渠道”开关。
- 一个渠道可以属于多个分组。
- 管理员不手动填写渠道别名。
- 用户通过选择控件配置，不手动输入别名。
- 配置入口放在个人菜单下，新增“渠道偏好”子菜单。

## 2. 用户界面

在个人菜单下新增“渠道偏好”页面，页面按 API 密钥创建页面可选的实际分组展示配置行：

```text
渠道偏好

分组          首选渠道
default       [ default-A7K2XP ▾ ]
vip           [ 跟随系统       ▾ ]
pro           [ pro-Q9M4LK     ▾ ]

                         [ 保存 ]
```

页面规则：

- 分组来源复用创建 API 密钥时使用的用户分组接口（当前为用户可用分组接口）。
- 排除 `auto`。
- 每个分组选项只包含该分组下当前启用的渠道。
- 选项只展示 `分组名-六位别名`，不展示真实渠道 ID、渠道名称、类型和上游地址。
- “跟随系统”表示清除该分组的首选渠道配置。
- 渠道被禁用或移出分组后，不再出现在选项中，运行时忽略对应的旧配置。

用户配置内部保存真实渠道 ID，前端只处理别名。

## 3. 路由优先级

对非 `auto` 分组，选择顺序为：

```text
用户首选渠道（可用时）
    ↓ 不存在或不可用
现有渠道亲和性
    ↓ 未命中
管理员全局渠道选择
```

首选渠道必须同时满足：

- 渠道处于启用状态；
- 渠道当前属于请求分组；
- 渠道支持请求模型；
- 渠道支持当前请求路径（适用于有路径限制的渠道类型）。

用户首选配置优先于现有亲和性，因为亲和性是管理员规则且不是用户级配置。未配置首选渠道的用户保持现有亲和性行为不变。

`auto` 分组完全跳过用户首选配置，继续使用现有自动分组逻辑。

## 4. 两层重试语义

单渠道重试和全局渠道切换是两层独立机制：

```text
选中渠道
    ↓ 请求失败
按该渠道 RetryTimes 重试同一渠道
    ↓ 该渠道重试耗尽
按全局 RetryTimes 回到管理员全局渠道选择
```

首选渠道作为首次选中的渠道，并完整遵守自己的 `RetryTimes`。该渠道重试耗尽后，才进入管理员全局渠道选择。

进入全局选择时：

- 直接进入现有外层下一次重试；
- 继续使用管理员现有优先级、权重和跨分组规则，不重置重试索引；
- 不额外排除首选渠道，因此管理员全局规则可能再次选中它；
- 全局 `RetryTimes = 0` 时不切换渠道，但单渠道自身重试仍然有效。

现有代码中，外层循环使用全局 `common.RetryTimes`，内层循环使用 `channel.RetryTimes`。

## 5. 现有亲和性

现有亲和性由管理员规则控制，缓存键结合规则、模型、分组和请求亲和键值；它不是用户级配置。

当前行为：

1. 首次选择时尝试命中亲和性渠道。
2. 亲和性渠道可用时直接使用。
3. 请求成功后按现有设置写回亲和性缓存。
4. 渠道失败后的外层重试重新执行全局渠道选择。
5. `skip_retry_on_failure` 只应对真正命中亲和性渠道的失败生效，不能阻止首选渠道失败后的正常重试。

配置首选渠道后，只改变该用户该分组的首次选择顺序，不修改亲和性缓存结构和管理员规则。

## 6. 渠道别名

展示名称为：

```text
分组名 + "-" + 六位别名
```

同一个渠道在不同分组下应使用不可关联的别名。推荐使用完整分组名作为格式保留加密的 `tweak`：

```text
alias = FPE(
    key = 专用稳定密钥,
    tweak = Hash(完整且规范化的分组名),
    plaintext = Base36(channel_id，补齐到 6 位),
)
```

示例：

```text
渠道 ID 12
default 分组 → default-A7K2XP
vip 分组     → vip-Q9M4LK
```

不能只使用分组名第一个字符，否则 `default` 和 `dev` 等分组无法可靠区分。

### 6.1 算法要求

- 使用 `github.com/capitalone/fpe/ff1`（v1.2.1）实现 FF1，radix 使用该库支持的最大值 36。
- 字符集固定为 `0-9A-Z`；加密库内部使用小写，接口展示时转为大写，解密前转回小写。
- 不自行发明密码算法。
- 不能把普通 AES 密文截断成 6 位，截断会失去可逆性并产生碰撞风险。
- `nanoid` 是随机 ID，不具备解密能力，不能直接作为可逆加密方案。
- 不需要“先加密 ID，再整体加密”的二次加密；FPE 已经提供固定长度的确定性可逆映射。
- Base36 六位域容量为 `36^6 = 2,176,782,336`，覆盖正向 int32 渠道 ID；编码前必须验证 `0 < channel_id < 36^6`，不能静默截断。
- 使用独立、稳定、可配置的服务端密钥，不能硬编码；密钥轮换会改变别名，需要后续定义版本策略。

别名只用于展示和配置接口，不承担权限控制。

## 7. 多分组和渠道迁移

不建立别名映射表，别名根据当前分组和渠道 ID 实时计算，不需要旧渠道回填。

渠道从 `default` 转移到 `vip` 时：

```text
转移前：default-A7K2XP
转移后：vip-Q9M4LK
```

处理规则：

- 新分组下自动计算新别名；
- 旧分组下的名称立即失效；
- 用户首选配置内部保存真实渠道 ID，不因别名变化丢失；
- 运行时重新验证渠道当前是否属于请求分组；
- 用户原先在旧分组选择的渠道不会自动跟随到新分组；
- 如果渠道同时属于多个分组，每个分组独立计算别名。

分组重命名后旧分组配置失效，用户需要在新分组下重新选择。

## 8. 用户配置接口

用户设置内部建议保存为分组到渠道 ID 的映射：

```json
{
  "preferred_channels": {
    "default": 12,
    "vip": 35
  }
}
```

接口只接收结构化的分组和别名，不要求后端从展示字符串拆分分组名：

```json
{
  "group": "vip",
  "channel_alias": "Q9M4LK"
}
```

保存时后端：

1. 使用分组和密钥解密别名得到渠道 ID；
2. 验证渠道存在且启用；
3. 验证渠道当前属于该分组；
4. 验证用户有权使用该分组；
5. 保存真实渠道 ID，不保存展示别名。

现有用户设置更新接口会重建整个 `UserSetting`，新增 `preferred_channels` 后必须保留该字段，避免修改通知等其他设置时覆盖渠道配置。

## 9. 用户请求日志

现有日志已经保存请求分组和渠道 ID：

```text
Log.Group
Log.ChannelId
```

因此不需要新增日志表字段，也不需要重新保存历史日志。用户日志查询响应阶段可以根据已有字段动态计算：

```text
日志 Group + ChannelId
    ↓
FPE 计算别名
    ↓
default-A7K2XP
```

管理员日志继续显示真实渠道名称和渠道 ID；普通用户日志应：

- 返回 `channel_display_name`，例如 `default-A7K2XP`；
- 不返回原始渠道 ID；
- 不返回真实渠道名称、上游地址或渠道类型；
- 保持现有管理员日志字段和权限行为不变。

现有管理员日志的 `Other.admin_info.use_channel` 已保存渠道重试链路。第一版普通用户只显示最终使用渠道，不新增日志存储。若将来要显示完整链路，需要同时记录每次尝试对应的分组；当前 `use_channel` 只有渠道 ID，在 `auto` 场景无法准确还原每次尝试的分组。

历史日志的分组和渠道 ID 仍然存在，因此可以动态生成当时的别名；密钥必须保持稳定，否则历史别名会发生变化。

## 10. 实际实现方案

### 10.1 实施顺序

```text
稳定密钥与 FF1 别名工具
    ↓
用户偏好存储和查询/保存接口
    ↓
首选渠道接入首次选择和重试链路
    ↓
普通用户日志别名化与脱敏
    ↓
个人菜单、配置页面和日志展示
    ↓
后端测试、前端检查和视觉验证
```

按这个顺序实施可以先固定后端契约，再接前端，避免页面依赖未稳定的别名格式。

### 10.2 稳定密钥

从环境变量 `CHANNEL_ALIAS_KEY` 读取密钥。

密钥处理规则：

- 使用 AES-128 密钥，即 16 个随机字节，以 Base64 形式配置。
- 环境变量缺失或格式错误时直接拒绝启动。
- 密钥不自动生成、不写数据库、不返回前端、不写日志。
- 多实例必须配置相同密钥；更换或丢失密钥会改变全部历史别名和当前别名。
- 第一版不提供密钥轮换入口，避免同一渠道历史显示名称变化。

密钥在服务启动时读取；初始化失败应阻止启动，不能退化为明文渠道 ID。

### 10.3 别名服务

新增 `service/channel_alias.go`，只暴露稳定业务能力：

```go
func EncryptChannelAlias(group string, channelID int) (string, error)
func DecryptChannelAlias(group string, alias string) (int, error)
```

实现规则：

1. 分组名使用后端返回的规范名称，仅做 `strings.TrimSpace`，不转换大小写。
2. 使用 `sha256.Sum256([]byte(group))` 的前 16 字节作为 FF1 tweak。
3. 把渠道 ID 转为固定六位 Base36 小写字符串。
4. 使用 `ff1.NewCipher(36, 16, key, tweak)` 加密。
5. 对外别名转为大写；解密时先校验长度和字符集，再转为小写。
6. 解密后验证结果是正整数且小于 `36^6`。

相同分组和渠道 ID 必须稳定得到相同别名；同一渠道在不同分组下应得到不可关联的结果。

### 10.4 用户设置存储

在 `dto/user_settings.go` 的 `UserSetting` 中增加：

```go
PreferredChannels map[string]int `json:"preferred_channels,omitempty"`
```

保存规则：

- map 的 key 是实际分组名，value 是真实渠道 ID。
- 不保存 `auto`。
- 专用保存接口先读取当前 `UserSetting`，只替换 `PreferredChannels`，再调用现有 `model.UpdateUserSetting`，不得覆盖通知、语言、扣费偏好等其他字段。
- 现有 `controller.UpdateUserSetting` 重建设置对象时，也必须从旧设置复制 `PreferredChannels`。
- 分组重命名或渠道移出分组时不批量清理用户设置，运行时验证失败后按未配置处理。

不新增用户偏好数据表，也不修改数据库结构。

### 10.5 用户接口

在 `router/api-router.go` 的用户鉴权路由下增加：

```text
GET /api/user/self/channel_preferences
PUT /api/user/self/channel_preferences
```

建议新增 `controller/channel_preference.go`，避免继续扩大 `controller/user.go`。

GET 响应示例：

```json
{
  "success": true,
  "data": {
    "groups": [
      {
        "group": "default",
        "selected_alias": "A7K2XP",
        "channels": [
          {
            "alias": "A7K2XP",
            "display_name": "default-A7K2XP"
          }
        ]
      }
    ]
  }
}
```

GET 实现：

1. 使用与 `/api/user/self/groups` 相同的 `service.GetUserUsableGroups` 和分组倍率配置，只保留创建 API 密钥时实际返回的分组。
2. 删除 `auto`。
3. 使用 GORM 只查询启用渠道的 `id` 和 `group` 字段，在 Go 中拆分渠道的多分组字符串。
4. 只把渠道加入用户可用分组，按别名排序，响应中不出现真实渠道 ID。
5. 对已保存但已禁用、已移出分组的渠道返回空 `selected_alias`，页面显示“跟随系统”。

PUT 使用整页覆盖，避免逐行接口和部分成功：

```json
{
  "preferences": {
    "default": "A7K2XP",
    "vip": ""
  }
}
```

PUT 实现：

1. 校验所有分组都是用户当前可用分组，拒绝 `auto`。
2. 空字符串表示删除该分组配置。
3. 按分组解密别名得到渠道 ID。
4. 验证渠道存在、启用并当前属于该分组。
5. 所有项验证通过后一次性更新用户设置；任一项失败则整次请求失败。

### 10.6 首选渠道接入路由

修改 `middleware/distributor.go` 的首次渠道选择顺序：

1. 从 `ContextKeyUserSetting` 读取 `PreferredChannels`。
2. 当前 using group 为 `auto` 时跳过。
3. 找到当前分组的首选渠道 ID 后，从渠道缓存读取渠道。
4. 使用现有状态、`model.IsChannelEnabledForGroupModel` 和请求路径检查验证可用性。
5. 可用时直接作为首次渠道。
6. 不可用或未配置时，继续执行现有亲和性，再执行管理员全局选择。

用户首选渠道被使用时，不提前初始化亲和性上下文，避免亲和规则的 `skip_retry_on_failure` 错误作用于用户首选渠道。首选渠道没有使用时，亲和性逻辑保持原样。

### 10.7 单渠道重试和全局切换

`controller/relay.go` 的内层 `channelRetry` 循环保持不变，因此首选渠道继续完整执行自己的 `channel.RetryTimes`。

首选渠道重试耗尽并进入外层全局重试时：

- 完全复用现有 `RetryParam` 和下一重试索引。
- 不维护已尝试渠道集合，也不重置优先级；管理员规则可能再次选中首选渠道。
- `auto` 跨分组重试继续使用现有索引状态，不叠加用户首选逻辑。

首选渠道只占首次请求位置，不改变全局 `common.RetryTimes` 的含义：全局次数仍表示首次渠道失败后最多执行多少次渠道级 fallback。

### 10.8 普通用户日志响应

现有 `Log.Group` 和 `Log.ChannelId` 已满足最终渠道别名计算，不修改日志表和日志写入链路。

实现时使用嵌入 `model.Log` 的轻量普通用户响应：

- 保留现有用户可见字段。
- 将原始 `channel`/渠道 ID 置零并清空 `channel_name`。
- 增加 `channel_display_name`。
- 在 controller 返回前根据日志已有的 group 和 channel ID 计算别名。
- 管理员 `GetAllLogs` 继续返回原结构。

必须同时检查 `GetUserLogs`、`GetLogByTokenId` 以及其他普通用户可访问的日志接口，保证没有路径继续泄露真实渠道 ID。

第一版只展示最终渠道。现有 `Other.admin_info.use_channel` 继续仅管理员可见，不新增重试链路存储。

普通用户日志只展示匿名渠道，不提供按匿名渠道筛选。

### 10.9 前端页面和菜单

在 `web/default` 增加：

```text
src/features/channel-preferences/
  api.ts
  types.ts
  index.tsx
src/routes/_authenticated/channel-preferences/index.tsx
```

菜单接入：

- 在桌面个人下拉菜单和移动端个人菜单增加同一路由入口。

页面实现：

- 使用 React Query 获取和保存配置。
- 每个分组一行，使用现有设计系统的可搜索 Combobox；渠道少时仍保持同一控件，避免数量变化导致交互切换。
- 选项包含“跟随系统”和该分组的匿名渠道名称。
- 使用一个页面级保存按钮批量提交，保存成功后刷新查询缓存。
- 加载时使用稳定高度的 Skeleton，空分组时显示现有 Empty 组件。
- 不在页面添加解释性文案。

### 10.10 前端日志展示

修改普通请求日志的类型和列：

- `features/usage-logs` 增加 `channel_display_name` 字段。
- `common-logs-columns.tsx` 中管理员继续显示真实渠道；普通用户显示 `channel_display_name` 的中性徽章。
- `details-dialog.tsx` 普通用户只显示匿名渠道名称。
- 普通用户不提供渠道筛选；管理员筛选栏保持渠道 ID 行为不变。

用户日志接口不能把加密密钥或原始渠道 ID 交给前端，所有加解密都在后端完成。

### 10.11 国际化

新增页面、菜单、按钮、空状态和错误信息必须使用 i18n。实现时按项目 i18n 流程同步以下语言：

```text
en, zh, zh-TW, fr, ja, ru, vi
```

前端文案使用 `useTranslation()`，后端参数错误使用现有 i18n 错误响应模式。

## 11. 测试和验收

### 11.1 后端测试

别名测试：

- 相同 group + channel ID 输出稳定。
- 不同 group 的同一渠道输出不可关联。
- 加密后固定六位且只包含 `0-9A-Z`。
- 加密再解密得到原渠道 ID。
- 非法长度、非法字符、空分组、非正 ID 和超出 `36^6` 拒绝。
- 密钥缺失或格式错误不会回退为明文。

接口测试：

- 只返回 API 密钥页面可用的非 `auto` 分组。
- 只返回启用渠道，且不泄露真实渠道 ID。
- 同一渠道在多个分组返回不同别名。
- 保存时拒绝无权分组、`auto`、禁用渠道和跨分组别名。
- 更新渠道偏好不会覆盖其他用户设置。

路由测试：

- 可用首选渠道优先于亲和性和全局选择。
- 首选渠道不可用时继续现有亲和性。
- `auto` 不读取首选配置。
- 单渠道 `RetryTimes` 耗尽后才切换。
- fallback 完全沿用现有重试索引和管理员规则。

日志测试：

- 管理员日志仍返回真实渠道 ID 和名称。
- 普通用户日志只返回别名，不返回真实渠道 ID 和名称。
- 旧日志可以通过已有 group + channel ID 生成别名。

新增或重写的 Go 测试使用 `testify/require` 和 `testify/assert`。

- 分组与 API 密钥创建页面一致且没有 `auto`。
- 多分组渠道显示不同别名。
- 保存、清除为“跟随系统”、刷新回显正确。
- 普通用户日志列表和详情只显示匿名渠道。
- 管理员日志显示和筛选行为不变。
- 使用 Playwright 检查桌面和移动端布局，没有文本溢出、控件重叠和不可操作状态。

### 11.3 完成条件

只有以下条件全部满足才算实现完成：

1. 用户可以从个人菜单配置每个非 `auto` 分组的首选渠道。
2. 请求首次选择、亲和性、单渠道重试和全局 fallback 符合本文顺序。
3. 普通用户在任何日志接口中都看不到真实渠道 ID 和名称。
4. 普通用户日志能稳定看到最终渠道匿名名称。
5. 管理员现有渠道配置、日志和路由行为没有回归。
6. 后端测试、前端类型检查、lint、构建和桌面/移动端视觉检查全部通过。
